# ask-cto — Independent Technical Advisor

codex-architect를 프레시한 컨텍스트로 소환하여 객관적인 기술 조언을 받는 커맨드.
현재 대화의 편향 없이 코드베이스를 직접 읽고 판단한다.

## Workflow

### Step 1: 질문 정리

`$ARGUMENTS`에서 사용자의 질문을 파악한다. 비어있으면 한 번만 확인한다:

- 어떤 코드/설계에 대한 의견인지
- 어떤 관점의 조언이 필요한지 (구조, 성능, 유지보수성, 확장성 등)
- 관련 파일/디렉토리 경로

### Step 2: 컨텍스트 수집

질문에 필요한 최소한의 컨텍스트를 정리한다:

- 관련 파일 경로 목록
- 현재 구조의 핵심 요약 (3-5줄)
- 구체적인 질문 또는 검토 요청
- 고려 중인 대안이 있으면 포함

### Step 3: codex-architect 소환

Agent 도구로 `codex-architect` 서브에이전트를 디스패치한다.
**반드시 프레시한 컨텍스트로** — 현재 대화의 결론이나 편향을 주입하지 않는다.

프롬프트 템플릿:

```
역할: 독립적인 CTO/아키텍트로서 객관적 기술 조언을 제공한다.

## 배경
<현재 구조의 핵심 요약>

## 관련 파일
<파일 경로 목록>

## 질문
<사용자의 구체적인 질문>

## 요청
1. 관련 코드를 직접 읽고 현재 구조를 파악하라.
2. 질문에 대해 객관적으로 판단하라.
3. 결론 + 근거 + 대안(있으면)을 제시하라.
4. 한국어로 답변. 간결하게.

## 제약
- 코드를 직접 읽고 판단. 추측 금지.
- 불필요한 패턴 도입 금지. 현재 패턴 존중.
- 응답은 500자 이내로 핵심만.
```

서브에이전트 설정:
- `subagent_type`: `lstack:codex-architect`
- `model`: 생략 (inherit)
- 포그라운드 실행 (결과를 바로 전달해야 하므로)

### Step 4: 결과 전달

codex-architect의 답변을 사용자에게 그대로 전달한다.
PM의 해석이나 편집을 최소화 — 독립적인 의견의 가치를 보존한다.

필요하면 짧게 보충:
- PM의 기존 생각과 다른 점
- 추가 논의가 필요한 부분

## Anti-patterns

- 현재 대화의 결론을 프롬프트에 넣어 답을 유도하지 않는다
- codex-architect의 답변을 임의로 수정하거나 필터링하지 않는다
- 단순한 코드 질문에는 이 커맨드를 쓰지 않는다 — 설계/구조 수준의 판단에만 사용
