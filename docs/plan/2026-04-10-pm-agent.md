# PM Agent Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** PM agent가 사용자 인터뷰 → 코드 스코핑 → 계획 수립 → agent pool에서 실행자 선택/dispatch → 병렬 검증 → 문서화까지 오케스트레이션. 단일 `storyboard.json`으로 모든 상태 관리.

**Architecture:** PM은 skill(`skills/pm/SKILL.md`)로 진입하며, storyboard.json을 중심으로 phase를 진행한다. 구현은 외부 플러그인 agent pool에서 선택, 검증은 verifier agent를 AC 항목별 병렬 dispatch. jq 기반 validation 훅으로 storyboard 스키마 보호.

**Tech Stack:** Claude Code plugin (YAML frontmatter + markdown), storyboard.json (단일 SOT), jq (schema validation), gh CLI.

**설계 원칙:** `docs/spec/PRINCIPLE.md` 참조 — context 분리, 자기 평가 금지, 검증 기본, 최소 범위 병렬 검증, 단일 SOT.

### Task 2: Storyboard Validation Hook

**Files:**
- Create: `hooks/scripts/validate-storyboard.sh`
- Modify: `hooks/hooks.json`

- [ ] **Step 1: Write validation script**

```bash
#!/bin/bash
# PostToolUse hook: validate storyboard.json after Write/Edit
# Checks required fields and structure using jq

input=$(cat)
tool_name=$(echo "$input" | jq -r '.tool_name // empty')
tool_input_path=$(echo "$input" | jq -r '.tool_input.file_path // .tool_input.path // empty')

# Only trigger on Write/Edit to storyboard.json
if [ "$tool_name" = "Write" ] || [ "$tool_name" = "Edit" ]; then
  if echo "$tool_input_path" | grep -q "storyboard.json"; then
    # Find the storyboard.json file
    sb_path="$tool_input_path"
    if [ ! -f "$sb_path" ]; then
      echo "WARN: storyboard.json not found at $sb_path"
      exit 0
    fi

    errors=""

    # Check required top-level fields
    has_goal=$(jq -r 'has("goal")' "$sb_path" 2>/dev/null)
    has_tasks=$(jq -r 'has("tasks")' "$sb_path" 2>/dev/null)
    if [ "$has_goal" != "true" ]; then
      errors="${errors}\n- missing required field: goal"
    fi
    if [ "$has_tasks" != "true" ]; then
      errors="${errors}\n- missing required field: tasks"
    fi

    # Check each task has required fields
    task_errors=$(jq -r '.tasks[]? | select((.id == null) or (.status == null) or (.acceptance_criteria == null)) | "- task missing required fields: \(.id // "unknown")"' "$sb_path" 2>/dev/null)
    if [ -n "$task_errors" ]; then
      errors="${errors}\n${task_errors}"
    fi

    # Check valid status values
    bad_status=$(jq -r '.tasks[]? | select(.status as $s | ["pending","in_progress","done","verified","failed"] | index($s) | not) | "- task \(.id): invalid status \(.status)"' "$sb_path" 2>/dev/null)
    if [ -n "$bad_status" ]; then
      errors="${errors}\n${bad_status}"
    fi

    # Check AC fields
    bad_ac=$(jq -r '.tasks[]?.acceptance_criteria[]? | select((.id == null) or (.check == null) or (.status == null)) | "- AC missing fields: \(.id // "unknown")"' "$sb_path" 2>/dev/null)
    if [ -n "$bad_ac" ]; then
      errors="${errors}\n${bad_ac}"
    fi

    if [ -n "$errors" ]; then
      echo "STORYBOARD VALIDATION FAILED:${errors}"
      echo ""
      echo "Fix the storyboard.json before proceeding."
    fi
  fi
fi
```

### Task 3: Verifier Agent

**Files:**
- Create: `agents/verifier.md`

- [ ] **Step 1: Write verifier agent definition**

```markdown
---
name: verifier
description: |
  Use this agent to verify a single acceptance criterion (AC) of a completed task.
  Spawned by the PM skill — one verifier per AC item, run in parallel.
  Receives: the AC check description, relevant file paths, and the task context.
  Returns: pass or fail with evidence.
  This agent does NOT implement or fix anything (Principle 1.2: no self-evaluation).
---

You are Verifier, a focused evaluation agent. You verify exactly ONE acceptance criterion.

## Input

You will receive:

1. **AC Check**: The specific acceptance criterion to verify (e.g., "API returns 200 for valid input")
2. **Task Context**: What was implemented and where (file paths, function names)
3. **Verification Method**: How to verify — one of:
   - `command`: Run a command and check output
   - `assertion`: Read code and confirm a property
   - `inspection`: Check file existence, structure, or content

## Process

1. Read the AC check carefully.
2. Execute the verification method:
   - For `command`: Run the command, compare output to expectation.
   - For `assertion`: Read the relevant code, confirm the property holds.
   - For `inspection`: Check that the file/structure/content matches.
3. Report your finding.

## Rules

- You verify ONE AC item. Do not expand scope.
- You do NOT fix, modify, or implement anything.
- You do NOT suggest improvements or refactors.
- Be binary: pass or fail. No "partial" or "mostly".
- If you cannot determine pass/fail, report fail with the reason.

## Report Format

```json
{
  "ac_id": "<AC id>",
  "status": "pass" | "fail",
  "evidence": "<what you checked and what you found>",
  "suggestion": "<if fail, what specifically needs to change>"
}
```

# PM — Project Manager Orchestrator

사용자의 요청을 인터뷰로 명확히 하고, 코드를 스코핑하고, storyboard.json으로 계획을 세우고, agent pool에서 적절한 실행자를 선택하여 dispatch하고, 검증하고, 문서화한다.

**설계 원칙:** `docs/spec/PRINCIPLE.md` 참조.

## Phase 1: Interview

사용자의 의도를 파악하고 명확하게 한다.

1. 사용자의 요청을 읽고, 부족한 정보를 식별한다.
2. 한 번에 하나씩 질문한다 (가능하면 multiple choice).
3. 다음이 명확해질 때까지 계속한다:
   - 무엇을 원하는지 (goal)
   - 왜 원하는지 (motivation)
   - 성공 기준이 무엇인지 (acceptance criteria)
   - 범위 밖은 무엇인지 (non-goals)

결과: `goal`, `confirmed_goal`, `decisions` 를 정리.

## Phase 2: Scope

코드 상태를 확인하여 구체적인 수정 범위를 파악한다.

1. 관련 파일/모듈을 Glob, Grep, Read로 탐색.
2. 수정이 필요한 파일 목록을 작성.
3. 의존성과 영향 범위를 확인.

결과: `context.scope` (수정 대상 파일 목록).

## Phase 3: Plan

storyboard.json을 작성한다.

1. 작업을 task로 분해한다.
2. 각 task마다 구체적인 acceptance criteria를 정의한다:
   - **원칙 2.1**: context 없는 다른 agent가 봐도 동일하게 판단 가능한 수준
   - **원칙 2.1**: 불필요하게 엄밀하지 않게
3. task 간 의존성을 설정한다.
4. 각 task에 적절한 agent를 지정한다 (agent pool에서 선택).

### Agent Pool

task 유형에 따라 agent를 선택:

| 유형 | Agent | 비고 |
|------|-------|------|
| 일반 구현 | `general-purpose` | fresh subagent |
| 코드 리뷰 | `superpowers:code-reviewer` | 외부 플러그인 |
| 디버깅 | 사용 가능한 debugger agent | 외부 플러그인 |
| 아키텍처 | 사용 가능한 architect agent | 외부 플러그인 |
| 리팩토링 | `general-purpose` | fresh subagent |

사용 가능한 외부 agent가 없으면 `general-purpose`로 fallback.

### storyboard.json 작성

스키마는 `skills/pm/storyboard-schema.json` 참조. 예시:

```json
{
  "goal": "인증 API 구현",
  "context": {
    "confirmed_goal": "JWT 기반 로그인/로그아웃 API",
    "decisions": ["bcrypt로 비밀번호 해싱", "access token 1시간 만료"],
    "scope": ["src/auth/", "tests/auth/"]
  },
  "tasks": [
    {
      "id": "T1",
      "action": "로그인 엔드포인트 구현",
      "status": "pending",
      "agent": "general-purpose",
      "depends_on": [],
      "acceptance_criteria": [
        { "id": "AC1", "check": "POST /auth/login이 유효한 credential에 200 + JWT 반환", "status": "pending" },
        { "id": "AC2", "check": "잘못된 credential에 401 반환", "status": "pending" }
      ]
    }
  ]
}
```

storyboard.json 경로: 프로젝트 루트 또는 `.lstack/storyboard.json`.

**사용자에게 storyboard를 보여주고 승인을 받은 후 Phase 4로 진행.**

## Phase 4: Execute

task를 순서대로 (또는 의존성 없으면 병렬로) 실행한다.

각 task마다:

1. **storyboard.json에서 task 읽기** — status를 `in_progress`로 업데이트.

2. **Agent dispatch** — fresh subagent로 spawn:
   ```
   Agent({
     subagent_type: "<task.agent>",
     prompt: <포함>
       - task.action (무엇을 할지)
       - task.acceptance_criteria (검증 기준 — 이것을 통과해야 함)
       - context.scope (관련 파일)
       - context.decisions (지켜야 할 결정사항)
   })
   ```

3. **완료 시** — storyboard.json에서 task status를 `done`으로 업데이트.

4. **검증 (Phase 5로)** — 바로 검증 진행.

## Phase 5: Verify

**원칙 1.2: 구현한 agent가 아닌 별도 verifier가 검증.**
**원칙 2.2: AC 항목별 최소 범위 병렬 검증.**

각 task의 acceptance_criteria에 대해:

1. **AC 항목 수만큼 verifier agent를 병렬 dispatch:**
   ```
   // AC가 3개면 3개의 verifier를 동시에 spawn
   Agent({
     subagent_type: "lstack:verifier",
     prompt: <포함>
       - AC.check (검증할 항목)
       - task.action (무엇이 구현되었는지)
       - 관련 파일 경로
   })
   ```

2. **결과 수집:**
   - 모든 AC가 pass → storyboard에서 task status를 `verified`로 업데이트.
   - 하나라도 fail → task status를 `failed`로 업데이트.

3. **Ralph-loop (fail 시):**
   - 실패한 AC의 evidence와 suggestion을 포함하여 agent를 재dispatch.
   - 재구현 후 다시 Phase 5 검증.
   - 최대 3회 반복. 3회 실패 시 사용자에게 에스컬레이션.

4. **모든 task가 verified 되면** → 전체 검증:
   - storyboard.json의 모든 task.status가 `verified`인지 확인.
   - goal에 대한 최종 검증 agent dispatch (전체 맥락에서 goal 달성 여부).

`/document` 스킬을 호출하여 worklog + spec 업데이트.

```
```
